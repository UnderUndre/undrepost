# Контекст для ревью (внешние ИИ)

Краткая цель

- Быстро дать внешнему рецензенту (ИИ) ясный контекст и чеклист для оценки артефактов фичи `001-init-from-payload`.
- Текущий статус: спецификация и план обновлены; проект-лид выбрал интеграцию аукционов как блокирующую (вариант A). В `tasks.md` добавлены T2.13..T2.16 (контракт/интеграция) и релизные гейты T4.4/T4.5. Анализатор (`reviews/analyze.md`) зафиксировал VERDICT: CRITICAL (требуется интеграция undrlla + контрактные тесты для релиза).

Ключевые артефакты (прочитать в первую очередь)

- `spec.md` — общий специ и решения (раздел "Sequencing consequence" содержит решение о блокирующих аукционах).
- `tasks.md` — полный список задач, dependency graph, Agent Summary (проверьте T2.13..T2.16, T4.4, T4.5).
- `plan.md` — план реализации и релиз-гейт описание.
- `data-model.md` — схемы коллекций (Polls, Reactions, Members и т.д.).
- `contracts/members.contract.md` — контракт JIT-provisioning.
- `reviews/analyze.md` — последний автоматический анализ и рекомендации.
- `quickstart.md` — локальный quickstart для воспроизведения окружения.

Область ревью — что именно проверить

1. Требования и покрытие

- Покрыты ли все ключевые FR-\* (особенно FR-030, FR-003, FR-002, FR-013)?
- Корректно ли привязаны задачи к требованиям (Task → FR mapping).

2. Архитектура аукционов (блокирующая)

- Убедиться, что `spec.md` явно помечает аукционы как блокирующие и что `tasks.md` содержит T2.13..T2.16.
- Проверьте dependency graph: T2.13..T2.16 должны стоять на критическом пути релиза.

3. Защита от хранения денег в контенте (FR-003)

- Проверьте, что есть `T4.4 Schema audit` и что в описании задачи есть шаги проверки схем и CI-прегейт (pre-merge).

4. Лицензирование (FR-002)

- Проверьте `T4.5 License & copyleft compliance` и короткую методику проверки runtime path.

5. SSE и хранение

- Проверьте в `plan.md` значения по умолчанию (heartbeat 15s, replay 100 events, caps) и в `tasks.md` задачи по infra (T2.3, T2.7, T2.8).

6. Тесты интеграции контрактов

- Убедитесь, что `T2.15` описывает e2e контрактные тесты с undrlla staging endpoint и что есть шаги для локального/CI запуска.

7. Agent tags / dispatch

- Убедитесь, что все задачи имеют `[AGENT]` теги и Agent Summary сопровождает распределение.

8. Острые сценарии и деградация

- Поведение виджета при недоступном undrlla (read-only), уникальность голосов/реакций при гонках, обработка expired polls.
